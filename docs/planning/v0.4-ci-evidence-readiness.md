# v0.4 CI Evidence Readiness Plan

## Objective

v0.4.0 should establish the CI Evidence Readiness Baseline for Guard-native Agent Harness.

The goal is to make the harness easier to validate in pull requests and CI without changing it into a SaaS, dashboard, hosted service, or runtime control plane. The CI flow should remain evidence-first, local, deterministic, CI-verifiable, and review artifact oriented, even when it runs inside GitHub Actions.

v0.4 planning is not approval, not enforcement, not autonomous execution, not a runtime control plane, and no authority grant. No provider output can authorize execution.

## Why This Follows v0.3

v0.3.0 established the Replayable Evidence Pack Baseline:

- v0.3 evidence pack contract.
- Deterministic evidence manifest rules.
- Fixture-based v0.3 evidence verification.
- Runtime generation of `evidence-manifest.json`.
- Runtime generated-pack verification.
- Deterministic local evidence inspection.
- JSON and Markdown inspection output.

CI evidence readiness is the next bounded step because v0.3 made evidence packs deterministic enough to validate repeatedly. v0.4 should focus on making those same local semantics easier to run and inspect in pull requests, without changing provider behavior, planner behavior, Tool Registry semantics, Policy Gate semantics, Guard Adapter semantics, or Guard runtime semantics.

## Tentative PR Sequence

This sequence is tentative and reviewable:

- PR 16: v0.4 CI evidence readiness plan.
- PR 17: GitHub Actions verification workflow.
- PR 18: CI generated evidence artifact smoke check.
- PR 19: v0.4 release gate and tag prep.

Each PR should preserve a narrow boundary and should avoid mixing CI readiness work with runtime feature changes.

PR 17 adds a deterministic GitHub Actions verification workflow and a local workflow-boundary verifier. It keeps CI evidence-first, local in semantics, deterministic, CI-verifiable, and review artifact oriented.

PR 18 adds a CI generated evidence artifact smoke check. It generates a minimal deterministic evidence pack with the existing mock runtime path, verifies the generated manifest, inspects the evidence pack in JSON and Markdown modes, and writes bounded review artifact outputs under `.artifacts/v0.4-ci-evidence-smoke/`.

## Strict Boundaries

v0.4 should not introduce:

- SaaS.
- Dashboard.
- OAuth.
- Database.
- Telemetry service.
- Pricing changes.
- License changes.
- Checkout or Paddle changes.
- API key persistence.
- `.env` loading.
- SDK dependency.
- Cloud model requirement.
- Autonomous execution expansion.
- Provider behavior changes.
- Planner behavior changes.
- Tool Registry semantic changes.
- Policy Gate semantic changes.
- Guard Adapter semantic changes.
- Guard runtime semantic changes.

CI should run existing local deterministic checks. It should not grant authority from a passing check, a provider response, a Guard result, or an uploaded artifact.

## Candidate CI Validation Commands

Candidate commands for a future GitHub Actions verification workflow:

```bash
npm install
npm run build
npm test
npm run lint
npm run verify:v0.1
npm run verify:v0.2:providers
npm run verify:v0.3:evidence
npm run verify:v0.3:runtime-evidence
npm run verify:v0.3:inspect-evidence
npm run verify:v0.3:release
npm run audit:summary
```

Full historical release-tag checks may need special handling in CI because tag-existence checks can behave differently after release. This planning PR does not change those scripts.

## Non-goals

This plan does not implement GitHub Actions, add CI workflows, upload artifacts, change runtime code, change package metadata, or prepare a release tag.

v0.4 planning does not require real provider calls, provider API keys, cloud models, hosted services, dashboards, OAuth, databases, telemetry services, commercial surfaces, or new SDK dependencies.

## Risks And Mitigations

Risk: CI tag state differs from local release-tag state.
Mitigation: Keep historical release-tag checks out of the first required CI workflow until their CI behavior is reviewed. Prefer CI-safe checks first.

Risk: Generated evidence artifacts make CI noisy or hard to inspect.
Mitigation: Start with deterministic verification commands, then add a separate smoke-check PR for generated evidence artifacts.

Risk: CI readiness is mistaken for deployment authority.
Mitigation: Preserve boundary language in docs and release checks: CI output is a review artifact, not approval, not enforcement, not autonomous execution, not a runtime control plane, and no authority grant.

Risk: Future workflow changes accidentally require provider credentials.
Mitigation: Keep the CI baseline on `mock` and local deterministic verification. Provider-specific checks should remain optional unless a future reviewed release changes that boundary.
