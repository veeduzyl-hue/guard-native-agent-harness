# v0.5 Evidence Review Profile Plan

## Objective

v0.5.0 should establish the Evidence Review Profile Baseline for Guard-native Agent Harness.

The goal is to define bounded review profiles that make evidence expectations easier to select, inspect, and discuss across local development, pull request CI, release preparation, and audit review. A review profile should describe expected verifier commands, required evidence files, inspector outputs, and review boundaries without changing runtime behavior or granting execution authority.

v0.5 planning remains evidence-first, local, deterministic, review profile oriented, and review artifact oriented. It is not approval, not enforcement, not autonomous execution, not a runtime control plane, and no authority grant. No provider output can authorize execution.

## Why This Follows v0.4

v0.4.0 established the CI Evidence Readiness Baseline:

- Deterministic GitHub Actions verification workflow.
- Full-history checkout for historical tag-aware baseline verification.
- `npm run verify:v0.4:ci-workflow`.
- CI generated evidence artifact smoke check.
- `npm run verify:v0.4:ci-evidence-artifact`.
- Bounded CI review artifact upload.
- `npm run verify:v0.4:release`.

Evidence review profiles are the next bounded step because v0.4 made local deterministic evidence checks repeatable in CI. v0.5 should make those evidence expectations clearer for different review contexts without changing provider behavior, planner behavior, Tool Registry semantics, Policy Gate semantics, Guard Adapter semantics, or Guard runtime semantics.

## Proposed Profile Types

This profile set is tentative and reviewable:

- `local-dev`: fast local review for developers iterating on docs, scripts, fixtures, or runtime behavior.
- `ci-pr`: pull request verification profile for deterministic CI checks and bounded review artifacts.
- `release-prep`: release-gate preparation profile for version, tag, release note, and evidence readiness checks.
- `audit-review`: deeper evidence inspection profile for review of generated evidence packs and their boundaries.

Profiles should be descriptive review inputs, not execution authority. A selected profile should tell reviewers what evidence was expected and what artifacts were produced. It should not approve work, enforce policy, authorize deployment, authorize merges, or permit runtime execution.

## Profile Controls

Each review profile should be able to describe:

- Which verifier commands are expected.
- Which evidence files are required.
- Which inspector outputs are produced.
- Which review boundaries are shown.

Candidate `local-dev` controls:

- Expected verifier commands: `npm run build`, `npm test`, `npm run lint`, and focused version-specific verifiers relevant to the local change.
- Required evidence files: fixture evidence or generated evidence only when the local change touches evidence behavior.
- Inspector outputs: local JSON or Markdown inspection output when evidence files are present.
- Review boundaries: local deterministic review only, no authority grant, no provider output can authorize execution.

Candidate `ci-pr` controls:

- Expected verifier commands: `npm install`, `npm run build`, `npm test`, `npm run lint`, `npm run verify:v0.4:ci-workflow`, `npm run verify:v0.4:ci-evidence-artifact`, and `npm run audit:summary`.
- Required evidence files: CI smoke evidence generated by existing deterministic mock paths.
- Inspector outputs: bounded CI review artifacts such as JSON and Markdown inspection output.
- Review boundaries: CI-verifiable review artifact only, not approval, not enforcement, not autonomous execution.

Candidate `release-prep` controls:

- Expected verifier commands: current release readiness checks, historical baseline checks, version checks, tag-prep checks, and release-gate verification.
- Required evidence files: release notes, tag-prep docs, final release gate docs, and deterministic evidence artifacts referenced by the release gate.
- Inspector outputs: release evidence inspection outputs needed for human review.
- Review boundaries: release preparation only, no tag creation before the final release gate, no GitHub Release publishing by profile selection.

Candidate `audit-review` controls:

- Expected verifier commands: evidence pack verification, runtime evidence verification, inspect-evidence verification, and relevant release readiness checks.
- Required evidence files: deterministic evidence manifest, events, final report, and bounded generated review artifacts.
- Inspector outputs: JSON and Markdown inspection output with clear boundaries and omitted secret material.
- Review boundaries: audit review artifact only, not certification, not deployment approval, not merge approval.

## Strict Boundaries

v0.5 should not introduce:

- SaaS.
- Dashboard.
- OAuth.
- Database.
- Telemetry service.
- Pricing changes.
- License changes.
- Checkout or Paddle changes.
- API key persistence.
- `.env` loading.
- SDK dependency.
- Cloud model requirement.
- Autonomous execution expansion.
- Provider behavior changes.
- Planner behavior changes.
- Tool Registry semantic changes.
- Policy Gate semantic changes.
- Guard Adapter semantic changes.
- Guard runtime semantic changes.
- Compliance certification.
- Deployment approval.
- Merge approval.

Profiles must not:

- Grant authority.
- Approve execution.
- Enforce policy.
- Become a runtime control plane.
- Authorize providers.
- Persist API keys.
- Load `.env` files.
- Require cloud models.
- Change how tools, policies, adapters, planners, providers, or the Guard runtime behave.

## Risks And Mitigations

Risk: Review profiles are mistaken for approval or enforcement.
Mitigation: Keep profile output framed as a review artifact. Repeat the boundary language: not approval, not enforcement, not autonomous execution, not a runtime control plane, and no authority grant.

Risk: Profiles become a hidden provider authorization layer.
Mitigation: Profiles must not authorize providers. No provider output can authorize execution.

Risk: Profile configuration drifts from existing verifier behavior.
Mitigation: Start with fixtures and schema checks before adding verifier behavior. Keep commands deterministic and tied to existing npm scripts.

Risk: Release and CI profiles create tag or release side effects.
Mitigation: Keep profile verification read-only. Do not add tag creation, tag push, release publishing, or package publishing commands.

Risk: Profile artifacts expose secrets or local machine details.
Mitigation: Keep artifacts bounded. Do not include `.env`, credentials, `node_modules`, full repositories, or unbounded generated directories.
